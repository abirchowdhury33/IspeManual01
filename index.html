9
Judge Q&A answers
Ready to use — no improvisation needed
  • Why AAS and not just a shared database or point-to-point API?
    A database forces both sides onto one system — rip-and-replace. A custom API is one integration that breaks when either side changes. AAS standardizes the data contract at the boundary. Rosemary keeps its MES, Roche keeps its QMS, they speak AAS at the handoff. New suppliers plug in with zero rework.
  • What if a small supplier doesn't have IT infrastructure for an API?
    AAS supports file-based exchange — AASX packages. A small lab generates a signed AASX file and sends it. We receive and parse it automatically. Same data model, zero custom integration. The benefit holds even without a live API connection.
  • How do you protect sensitive data — Rosemary can't see Roche's pricing?
    ABAC — Attribute-Based Access Control per AAS Security Part 4. Policies are scoped at submodel property level: role + asset ID + operation → allow or deny. Rosemary operators can only write to their own batch IDs. They cannot access Roche formulation data or pricing. Show the IAM tab — it's live in the simulation.
  • How does this maintain data integrity — better than a signed PDF?
    Cryptographically stronger. Every AASX package is signed with an X.509 certificate — tamper-evident and non-repudiable. SHA-256 hash per submodel write means any modification is detectable. A wet signature on a PDF can be forged. This cannot.
  • Is AAS mature enough for pharma production use?
    Yes. Catena-X runs AAS registry profiles in a live automotive production ecosystem today. The Digital Battery Passport uses AAS for EU regulatory compliance. IDTA and open-source runtimes like Eclipse BaSyx are production-grade. The standard is stable at IEC 63278-1.
  • How do you get Rosemary to adopt this?
    Their workload drops — no more generating PDFs, no more re-sending Excel files, no more chasing email confirmations. They get real-time demand signal from Roche SAP automatically. The adoption argument sells itself. We start with one batch type in pilot — low risk, high visibility.
  • How do regulators benefit?
    Any EMA, Swissmedic, or WHO inspector queries the AAS chain — full traceability from API synthesis to patient, every write with actor, timestamp, and hash. Retrievable in seconds instead of days of manual document assembly. We show this in the Traceability tab.

2-minute pitch
"Pharma supply chains depend on batch data arriving accurately and on time. Today, Rosemary completes a Basilmol DS batch — with all data digital in their MES — and then sends it to Roche as a PDF. A Roche QA employee reads it, compares serial numbers, and types it into the QMS. Meanwhile, Kaiseraugst sends a monthly demand forecast via email. Rosemary plans production on data that's already 1–2 weeks stale. The result: 4–6 weeks of batch cycle time, ±18% forecast error, and 3–5 manual handoffs per batch. For a first-in-class molecule, this directly delays patient access.

Our solution uses the Asset Administration Shell — an IEC/IDTA open standard — to eliminate this entirely. Each Basilmol DS batch gets an AAS shell with two submodels: a Batch CoA submodel and a Demand Forecast submodel. Rosemary's MES writes the CoA directly via REST API at the moment of release. Roche's QMS polls the AAS Registry and reads it automatically. And critically — it's bidirectional: Roche SAP writes the demand forecast back to Rosemary through the same AAS channel in real time.

This is ALCOA+ by design — not bolted on. The data is contemporaneous, original, attributable, and cryptographically signed with X.509. Any EMA or WHO inspector queries the full chain in seconds.

Our pilot runs in 90 days: one batch type, one supplier, prove the model. By Q4 the supply chain planner supervises exceptions — they don't enter data. And because AAS is a universal open standard, every future CDMO plugs in with zero rework."
